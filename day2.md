# HTTP / ネットワーク基礎 Day2 まとめ

# Day2の目的

Day1では：

「ブラウザが勝手にHTTP通信している」

を観察した。

Day2では：

「自分でHTTP通信を送る」

を目的として学習した。

特に：

・GET
・POST
・header
・JSON

の理解を行った。

--------------------------------------------------

# 使用したもの

## curl.exe

curl.exe

意味：

「HTTP通信を送るプログラム」

ブラウザなしで通信できる。

--------------------------------------------------

# なぜ curl.exe を使ったのか

PowerShellでは：

curl

が、

Invoke-WebRequest

へ置き換わる。

そのため：

curl.exe

で：

「本物のcurl」

を使った。

--------------------------------------------------

# curl の本質

curlは：

「通信専用プログラム」

ブラウザのように：

・HTML表示
・CSS
・JavaScript実行

はしない。

純粋な：

「HTTP通信だけ」

を行う。

--------------------------------------------------

# GET通信

実行：

curl.exe https://example.com

--------------------------------------------------

# GETとは

意味：

「データをください」

である。

--------------------------------------------------

# 起きたこと

curl
↓
example.comへHTTP通信
↓
HTML取得
↓
PowerShellへ表示

--------------------------------------------------

# 表示されたもの

<!doctype html>
<html>
...
</html>

これは：

「Webページの正体」

である。

ブラウザは普段これを：

・解釈
・レイアウト
・表示

しているだけ。

--------------------------------------------------

# 超重要理解

「Webページ
=
サーバーから送られてきたHTML」

である。

--------------------------------------------------

# Header確認

実行：

curl.exe -I https://example.com

--------------------------------------------------

# -I の意味

「headerだけ見せて」

--------------------------------------------------

# Headerとは

「HTTP通信の追加情報」

である。

--------------------------------------------------

# 見た内容

HTTP/1.1 200 OK

意味：

「HTTP通信成功」

--------------------------------------------------

# Content-Type

Content-Type: text/html

意味：

「返したデータはHTML」

--------------------------------------------------

# Server

Server: cloudflare

意味：

「Cloudflare経由」

--------------------------------------------------

# allow

allow: GET, HEAD

意味：

「許可しているHTTP method」

--------------------------------------------------

# 超重要理解

HTTP通信には：

「データ本体」

だけでなく：

「header情報」

も存在する。

--------------------------------------------------

# POST通信

実行：

curl.exe -X POST https://httpbin.org/post

--------------------------------------------------

# -X POST の意味

「POST通信を使う」

--------------------------------------------------

# POSTとは

意味：

「データ送信」

--------------------------------------------------

# 使用したサイト

https://httpbin.org/post

意味：

「送ったデータをそのまま返す練習用サーバー」

--------------------------------------------------

# /post の意味

「サーバー内の場所」

Laravelでいう：

/api/players

などと同じ。

--------------------------------------------------

# URL構造理解

例：

https://httpbin.org/post

分解：

## https://

通信ルール。

HTTPS通信。

--------------------------------------------------

## httpbin.org

通信先サーバー。

--------------------------------------------------

## /post

サーバー内URL。

--------------------------------------------------

# form形式送信

実行：

curl.exe -X POST https://httpbin.org/post -d "name=shugo"

--------------------------------------------------

# -d の意味

「送信データ」

--------------------------------------------------

# 送った内容

name=shugo

--------------------------------------------------

# form

表示：

"form": {
  "name": "shugo"
}

意味：

「サーバーが受け取ったデータ」

--------------------------------------------------

# form形式とは

昔ながらの：

「HTMLフォーム送信形式」

--------------------------------------------------

# Content-Type

application/x-www-form-urlencoded

意味：

「フォーム形式」

--------------------------------------------------

# JSON通信

実行：

curl.exe -X POST https://httpbin.org/post -H "Content-Type: application/json" -d '{\"name\":\"shugo\"}'

--------------------------------------------------

# -H の意味

「header追加」

--------------------------------------------------

# application/json

意味：

「JSON形式で送ります」

--------------------------------------------------

# JSON送信成功

表示：

"json": {
  "name": "shugo"
}

意味：

「JSONとして認識された」

--------------------------------------------------

# form と json の違い

## form

name=shugo

HTMLフォーム形式。

--------------------------------------------------

## json

{"name":"shugo"}

API向け通信形式。

React/Laravel APIで超重要。

--------------------------------------------------

# JSON.stringify()

Reactでよく使う：

JSON.stringify()

意味：

「JavaScriptオブジェクト
↓
JSON文字列へ変換」

--------------------------------------------------

# 例

元：

{
  name: "shugo"
}

変換後：

{"name":"shugo"}

--------------------------------------------------

# なぜ必要か

HTTP通信は基本：

「文字列」

を送るため。

--------------------------------------------------

# Reactとの繋がり

Reactの：

fetch(...)

は内部で：

「HTTP通信」

している。

--------------------------------------------------

# fetch例

fetch("/api/players", {
  method: "POST",
  headers: {
    "Content-Type": "application/json"
  },
  body: JSON.stringify({
    name: "shugo"
  })
})

--------------------------------------------------

# 超重要理解

HTTP通信構造：

HTTP
├ method
│   ├ GET
│   ├ POST
│   ├ PUT
│   └ DELETE
│
├ header
│   └ 追加情報
│
├ Content-Type
│   └ データ形式
│      ├ form
│      └ json
│
└ body
    └ 実際の送信データ

--------------------------------------------------

# Day2 最重要理解

HTTP通信は：

どこへ(URL)
↓
どんな目的(method)
↓
どんな形式(Content-Type)
↓
どんなデータ(body)

を送る通信

である。
