

# HTTP / ネットワーク基礎 Day3 完全まとめ

# Day3の目的

Day2では：

・GET
・POST
・body
・JSON
・form

など、

「HTTP通信で何を送るか」

を学んだ。

Day3では：

「通信に付属する追加情報」

である：

header

を深く理解した。

--------------------------------------------------

# headerとは

headerとは：

「HTTP通信に付く追加情報」

である。

HTTP通信は：

method
header
body

で構成される。

--------------------------------------------------

# HTTP通信の構造

HTTP
├ method
│   ├ GET
│   ├ POST
│   ├ PUT
│   └ DELETE
│
├ header
│   └ 通信の追加情報
│
└ body
    └ 実際のデータ

--------------------------------------------------

# method

methodは：

「何をしたい通信か」

を表す。

例：

GET
→ データ取得

POST
→ データ送信

PUT
→ 更新

DELETE
→ 削除

--------------------------------------------------

# body

bodyとは：

「実際の送信データ本体」

である。

例：

{"name":"shugo"}

--------------------------------------------------

# header

headerとは：

「通信の説明書」

である。

追加情報を複数持てる。

--------------------------------------------------

# Request Header と Response Header

HTTP通信は：

ブラウザ → サーバー
↓
サーバー → ブラウザ

の往復。

その両方にheaderが存在する。

--------------------------------------------------

## Request Header

ブラウザ → サーバー

へ送るheader。

例：

Accept
User-Agent
Authorization
Content-Type

--------------------------------------------------

## Response Header

サーバー → ブラウザ

から返るheader。

例：

Server
Allow
cf-cache-status

--------------------------------------------------

# Request URL

例：

https://example.com/

意味：

「どこへ通信したか」

--------------------------------------------------

# Request Method

例：

GET

「データをください」

--------------------------------------------------

# Status Code

例：

304 Not Modified

意味：

「変更なし
キャッシュを使って良い」

--------------------------------------------------

# Remote Address

例：

172.66.147.243:443

意味：

「実際の通信先IPアドレス:ポート番号」

--------------------------------------------------

# URL の構造

例：

https://example.com/api/players

分解：

--------------------------------------------------

## https

「通信ルール」

HTTPS通信。

--------------------------------------------------

## example.com

「ドメイン名」

人間向けサーバー名。

--------------------------------------------------

## /api/players

「サーバー内の場所」

LaravelではRouteへ繋がる。

--------------------------------------------------

# ドメインとは

例：

google.com
example.com
youtube.com

意味：

「人間向けの名前」

--------------------------------------------------

# IPアドレス

例：

172.66.147.243

意味：

「実際の通信先住所」

コンピュータは本来IPで通信している。

--------------------------------------------------

# DNS

DNSとは：

ドメイン名
↓
IPアドレス

へ変換する仕組み。

例：

example.com
↓
172.66.147.243

--------------------------------------------------

# ポート番号

例：

443
8000
3306

意味：

「サーバーの入口番号」

--------------------------------------------------

# よく使うポート

80
→ HTTP

443
→ HTTPS

3306
→ MySQL

22
→ SSH

8000
→ 開発サーバー

--------------------------------------------------

# User-Agent

例：

User-Agent: Chrome

意味：

「どのブラウザ・OS・端末から来たか」

サーバー側は：

Chrome用表示
スマホ用表示

などを切り替えられる。

--------------------------------------------------

# Accept

例：

Accept: application/json

意味：

「JSON形式で返してください」

Acceptは：

「受け取りたい形式」

を表す。

--------------------------------------------------

# Content-Type

例：

Content-Type: application/json

意味：

「送るデータはJSON形式です」

Content-Typeは：

「実際の送信データ形式」

を表す。

--------------------------------------------------

# Accept と Content-Type の違い

## Accept

「何を受け取りたいか」

--------------------------------------------------

## Content-Type

「何形式で送っているか」

--------------------------------------------------

# Authorization

例：

Authorization: Bearer xxx

意味：

「ログイン認証情報」

--------------------------------------------------

# Bearer token

ログイン成功
↓

サーバーがtoken発行
↓

以降：

Authorization: Bearer token

を毎回送る。

--------------------------------------------------

# サーバー側

token確認
↓
誰か判定
↓
許可 or 拒否

をしている。

--------------------------------------------------

# curl -H

実行：

curl.exe https://httpbin.org/headers -H "Test-Header: shugo"

--------------------------------------------------

# -H の意味

「header追加」

--------------------------------------------------

# 今回追加したheader

Test-Header: shugo

--------------------------------------------------

# 何が起きたか

HTTP通信のheader欄へ：

Test-Header: shugo

が追加された。

--------------------------------------------------

# httpbin.org/headers

意味：

「受け取ったheaderをそのまま返す練習用API」

--------------------------------------------------

# 実際の結果

"Test-Header": "shugo"

つまり：

「header追加成功」

である。

--------------------------------------------------

# Day3 最重要理解

HTTP通信は：

method
header
body

で構成される。

headerは：

「通信の追加情報」

であり、

Request Header と Response Header の両方が存在する。

また：

ドメイン
↓
DNS
↓
IPアドレス
↓
port

という流れで実際の通信が行われている。
